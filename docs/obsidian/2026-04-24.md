## What Thunderstrux Is Right Now

Thunderstrux is a Next.js App Router SaaS foundation for student societies with:

- Credentials auth
- Multi-tenant organisations
- Membership-based dashboard access
- Event creation/edit/publish flows
- Public event discovery
- Stripe Checkout purchases
- Stripe Connect onboarding
- Webhook-driven reconciliation

It is not yet a polished production app. It is a functional foundation with some current UI and DX rough edges that are now documented.

- `DashboardShell` hardcodes `Dashboard` as the active nav item. This is a real current-state limitation.
- `/dashboard/[orgSlug]/events/new` still renders both `Create Event` and the form title `New event`.
- Some page shells use `Card` while some newer pages use hand-written `rounded-xl border shadow-sm` wrappers. The app is visually more consistent than before, but not fully standardized.

## Data and Demo Behavior

Two separate mechanisms currently make the app look populated:

1. Real seed data in `prisma/seed.mjs`
2. Homepage fallback demo-event creation in `lib/events/public-events.ts`

This matters because:

- The homepage can show events even if auth users or memberships are missing
- The seed is still required for dashboard work

## Security Model

The backend generally does the right thing:

- Auth comes from session
- Membership and role are checked server-side
- `orgSlug` resolves to a real organisation plus membership row
- Event routes scope operations to organisation

Important nuance:

- Some helpers exist for `x-org-id` validation in `lib/db/organisation-scope.ts`
- The actual event create/update routes mainly trust validated `organisationId` and then verify access server-side
- Frontend-supplied org identifiers are inputs, not authority

## Event Rules That Matter

Publishing:

- Draft event cannot publish without at least one ticket type
- Total ticket quantity must be greater than zero

Editing:

- Ticket types with sales cannot be modified
- Ticket types with sales cannot be removed

Deletion:

- Events with orders or issued tickets cannot be deleted

Unpublishing:

- Published events with orders cannot be unpublished

## Payments and Stripe Reality

Checkout requires:

- Authenticated user
- Published event
- Valid ticket type
- Sufficient ticket quantity
- Organisation connected Stripe account
- Charges enabled on connected account

If checkout fails with `Stripe not connected`, the usual cause is:

- no `stripeAccountId`, or
- charges are not enabled yet

Stripe Connect settings page is the operational dashboard for this.

## Development Environment Reality

This project has already had repeated stale-build problems in Docker.

Current known-good approach:

- Docker Compose
- Webpack dev server
- polling enabled

Do not casually switch the dev script back to Turbopack.

## First Things To Check In A New Session

1. Is Docker Desktop running?
2. Is `docker compose up -d` healthy?
3. Does `http://localhost:3000` return `200`?
4. If UI is stale, does the container file match the host file?
5. If the homepage looks fine but dashboard is broken, has the seed been re-run?

## Practical Commands

Start stack:

```bash
docker compose up -d
```

Run migrations:

```bash
docker compose exec app pnpm prisma:migrate
```

Seed data:

```bash
docker compose exec app pnpm seed
```

Build:

```bash
docker compose exec app pnpm build
```
