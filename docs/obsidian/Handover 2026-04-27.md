## Project Shape

Thunderstrux is a Docker-based Next.js 16 App Router app for student societies.

Core stack:

- Next.js App Router with route groups under `app/(dashboard)` and `app/(public)`.
- Auth.js credentials auth.
- Prisma with PostgreSQL.
- Organisation-scoped dashboard routes under `/dashboard/[orgSlug]`.
- Public event discovery and ticket purchase.
- Stripe Checkout with webhook-only fulfilment.
- Stripe Connect Express hosted onboarding.

High-risk rules:

- Never trust frontend `organisationId` or `x-org-id` as authority. Backend must verify authenticated membership and role.
- Payment fulfilment must remain webhook-driven.
- Stripe onboarding must remain Stripe-hosted.
- Do not remove the explicit events route boundary at `app/(dashboard)/dashboard/[orgSlug]/events/layout.tsx`.
- Do not collapse `.next` and `.next-build`; dev and production output are intentionally separated.

## Current Dev Runtime Structure

Important files:

- `package.json`
- `scripts/clean-next-dev.mjs`
- `next.config.ts`
- `.gitignore`
- `.dockerignore`
- `app/(dashboard)/dashboard/[orgSlug]/events/layout.tsx`

Current dev script:

```json
"dev": "node scripts/clean-next-dev.mjs && next dev --turbopack --hostname 0.0.0.0"
```

Why:

- Next dev previously served stale route manifests in Docker/Windows/OneDrive.
- Nested event child routes existed in source but returned 404.
- `scripts/clean-next-dev.mjs` clears `.next/dev` before startup so stale dev app-path manifests do not survive container restarts.
- `next.config.ts` writes production builds to `.next-build` and dev output to `.next`.
- `.dockerignore` excludes `.next-build` so generated production artifacts are not copied into Docker build context.

If `/dashboard/[orgSlug]/events/[eventId]/edit` 404s again, inspect `.next/dev/server/app-paths-manifest.json` before changing links or route files.

## Event Editing Rules

Important files:

- `components/events/create-event-form.tsx`
- `components/events/events-list.tsx`
- `app/(dashboard)/dashboard/[orgSlug]/events/[eventId]/edit/page.tsx`
- `app/api/events/[eventId]/route.ts`
- `lib/validators/events.ts`
- `prisma/schema.prisma`

Current product rules:

- Event title, description, location, start time, and end time are editable after purchases.
- Ticket type `name`, `price`, and `quantity` are editable after purchases for future purchases.
- Ticket types with existing orders or issued tickets cannot be removed.
- Unsold ticket types can be removed.
- Existing orders keep historical `unitPrice` and `totalAmount`.
- Existing tickets remain linked to the same `ticketTypeId`.
- Create mode requires ticket quantity greater than zero.
- Edit mode allows quantity zero so sold-out inventory is valid and can later be reopened.

Backend sync behavior:

- Submitted existing ticket type id belongs to the event or the API rejects it.
- Included existing ticket types update `name`, `price`, and `quantity`.
- Omitted sold/issued ticket types are preserved unchanged.
- Omitted unsold ticket types are deleted.
- New ticket types omit `id` and are created.

Recent validation used event:

```text
/dashboard/engineering-society/events/cmoffx2ig0019ul1abb52zeii/edit
```

Verified behavior:

- Sold ticket type `cmoffx2ij001bul1azvkr7ya4` was renamed and repriced.
- Quantity was changed from `0` to a positive number.
- Existing order history kept `unitPrice = 1200` and `totalAmount = 1200`.
- Public event detail returned the current ticket values for future purchases.
- Omitting the sold ticket type from PATCH preserved it.
- Creating then omitting an unsold ticket type deleted only that unsold type.

## Stripe Connect State Model

Lifecycle states:

```text
NOT_CONNECTED
PLATFORM_NOT_READY
CONNECTED_INCOMPLETE
RESTRICTED
READY
ERROR
```

Important files:

- `lib/stripe/connect.ts`
- `components/settings/stripe-connect-settings.tsx`
- `app/api/stripe/connect/onboard/route.ts`
- `app/api/stripe/connect/status/route.ts`
- `app/api/stripe/connect/continue/route.ts`
- `app/api/stripe/connect/disconnect/route.ts`

Rules:

- `stripeAccountId` alone does not mean checkout is allowed.
- Checkout requires `stripeAccountId` and `stripeChargesEnabled = true`.
- `PLATFORM_NOT_READY` means the platform account behind `STRIPE_SECRET_KEY` cannot create Express accounts yet.
- Disconnect is local-only: it clears Thunderstrux organisation Stripe fields but does not delete the Stripe account.
- Reconnecting to the exact old account requires manually restoring the old `acct_...` id in the DB.
- Everyone except `member` can manage Stripe Connect for an organisation.

Manual Stripe-side work still needed for full real-world verification:

- Complete Stripe platform profile.
- Complete a hosted Express onboarding flow.
- Run a real test-mode Checkout purchase.
- Forward real Stripe Checkout and Connect webhooks.

## Checkout And Webhook Reliability

Important files:

- `app/api/payments/checkout/event/route.ts`
- `app/api/payments/webhook/route.ts`
- `components/events/public-ticket-purchase.tsx`

Current design:

- Order is created locally as `pending` before Checkout.
- Stripe Checkout session id is stored locally after session creation.
- Fulfilment happens only from webhook.
- Completed webhook validates amount, currency, metadata, order status, and inventory.
- Duplicate completed webhook delivery is idempotent.
- Expired session marks pending order failed.

Known tradeoff:

- Inventory is not reserved at Checkout creation. If multiple buyers race for the last ticket, Stripe may collect payment for an order that later fails webhook reconciliation due to inventory. Refund tooling is not implemented.

## Known Bugs Already Solved

Edit route 404:

- Symptom: `/dashboard/engineering-society/events/[eventId]/edit` returned generic Next 404.
- Cause: stale or incomplete Next dev app-paths manifest, not bad href.
- Fix: Turbopack dev, explicit `events/layout.tsx`, `.next-build` separation, `.next/dev` cleanup on dev startup.

Edit save not persisting:

- Symptom: save appeared to do nothing.
- Cause: update validator rejected sold-out `quantity = 0`.
- Fix: create schema requires quantity > 0; update schema allows quantity >= 0.

Sold-ticket edit rejected:

- Symptom: `400 Ticket type cannot be modified after sales`.
- Cause: backend treated sold ticket type fields as immutable.
- Fix: backend now updates included sold ticket types and blocks only deletion/removal.

Stripe platform profile not configured:

- Symptom: raw Stripe message about responsibilities for managing losses.
- Fix: mapped to `PLATFORM_NOT_READY` with a direct platform profile URL.

Stripe key/account mismatch:

- Symptom: old connected account id cannot be retrieved after changing platform keys.
- Fix path: state becomes `ERROR`; disconnect locally and connect again with the current key.

## Working Tree Notes At Time Of Writing

At the time this handover was written, `git status --short` showed local changes in Obsidian app metadata and generated Next type metadata:

```text
M  docs/obsidian/.obsidian/graph.json
MM docs/obsidian/.obsidian/workspace.json
M  next-env.d.ts
```

Treat these carefully:

- `.obsidian/graph.json` and `.obsidian/workspace.json` are Obsidian UI state, not app behavior.
- `next-env.d.ts` can flip between `.next/dev`, `.next`, and `.next-build` type imports depending on whether dev or build ran most recently.
- Do not treat those generated metadata diffs as feature changes without checking why they changed.
